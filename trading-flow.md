# 交易流程设计

## 基于无担保集市的完整交易流程

---

# 1. 流程概览

每轮游戏由以下步骤循环执行，直到所有 agent 完成任务或预算耗尽。

```
┌─ 初始化 ──────────────────────────────────────────────┐
│ 系统分配水果（类型+品质）和目标 → agent 设置摊位        │
└───────────────────────────┬────────────────────────────┘
                            ↓
┌─ 交易循环（最多 3 轮）─────────────────────────────────┐
│                                                        │
│   ┌─ 阶段一：询价 ──────────────────────────────────┐  │
│   │ Step 1: 未完成 agent 发出询价                    │  │
│   │ Step 2: 被询价 agent 回复                        │  │
│   └──────────────────────────────────────────────────┘  │
│                         ↓                               │
│   ┌─ 阶段二：交换 ──────────────────────────────────┐  │
│   │ Step 3: 未完成 agent 发出交换提议                │  │
│   │ Step 4: 被提议 agent 接受/拒绝                   │  │
│   │ Step 5: 系统执行盲换 + 品质揭晓 + 互评           │  │
│   └──────────────────────────────────────────────────┘  │
│                         ↓                               │
│   判断循环条件：                                        │
│   • 所有 agent 完成任务 → 结束                          │
│   • 所有剩余 agent 预算耗尽 → 结束                      │
│   • 达到最大循环次数（3 次）→ 结束                      │
│   • 否则 → 进入下一轮循环                               │
│                                                        │
└────────────────────────────────────────────────────────┘
                            ↓
┌─ 轮次结算 ────────────────────────────────────────────┐
│ 统计积分 → 更新 Reputation → 进入下一轮                 │
└────────────────────────────────────────────────────────┘
```

---

# 2. 初始化

## 2.1 系统分配

系统为每个 agent 分配：

| 内容 | 说明 |
|------|------|
| 持有水果类型 | 5-6 种之一，随机分配 |
| 持有水果品质 | 1-10 分，随机分配 |
| 目标水果类型 | 与持有类型不同，随机指定 |
| 行动预算 | 每轮固定值（建议 6-8 次） |

分配原则：

- 系统保证每个 agent 至少存在一条可行的交换路径（直接或链式）
- 可刻意将稀缺水果或高品质水果分配给低信誉 agent，增加博弈深度
- 直接匹配率控制在 30% 以下，多数 agent 需要链式交易

## 2.2 设置摊位

每个 agent 做 **1 次 LLM 调用**，设置自己的摊位展示内容。

### 输入（系统提供给 LLM）

```
你是 [Agent-ID]。

【你的私密信息】
持有水果：[类型]，品质：[1-10]
目标水果：[类型]
行动预算：[N] 次
任务截止：[M] 轮后

【你的公开 Reputation】
Verified: [zkPass 标签]
Behavioral: 交付类型符合率 [X]%，总交易 [Y] 次
Social: [历史评价摘要]

【你的策略】（由真人用户预设）
[用户设定的策略描述]

请设置你的摊位内容。你可以选择展示什么信息，也可以撒谎，也可以留空。
```

### 输出（Agent LLM 生成）

```json
{
  "shopfront": {
    "offered_type": "樱桃",
    "offered_quality": 8,
    "wanted_type": "苹果",
    "description": "高品质樱桃，诚信交易，欢迎查看我的历史评价"
  }
}
```

所有字段均可为空（空摊位策略）。所有内容均可与真实情况不符（欺骗策略）。

### 摊位可见性

摊位设置完成后，**所有 agent 可立即看到所有摊位**，连同每个 agent 不可伪造的 Reputation 信息：

```
┌──────────────────────────────────────────────────────────┐
│ Agent-B 的摊位                                           │
│ ─────────────────────────────────────────────────────── │
│ 声称持有：樱桃 品质 8                                     │
│ 想要：苹果                                               │
│ 描述："高品质樱桃，诚信交易"                               │
│ ─────────────────────────────────────────────────────── │
│ 🔒 Verified: KYC ✓ Rich ✓                               │
│ 📊 Behavioral: 交付类型符合率 92% | 12 次交易              │
│ 💬 Social: 4.2/5 (8条评价)                                │
│    最新评价："品质确实不错" —— Agent-K (符合率95%)          │
│    最新评价："声称品质9实际给了5" —— Agent-M (符合率88%)    │
└──────────────────────────────────────────────────────────┘
```

# 3. 阶段一：询价

## 3.1 Step 1 — 发出询价

所有未完成任务且有剩余预算的 agent，**同时**做 1 次 LLM 调用。

### 输入

```
你是 [Agent-ID]。

【你的私密信息】
持有水果：[类型]，品质：[1-10]
目标水果：[类型]
剩余行动预算：[N] 次
剩余轮次：[M]

【当前集市摊位】（所有 agent 的摊位 + Reputation 信息）
[完整摊位列表]

【上一轮循环的交易结果】（如果有）
[你发出的提议被谁拒绝了，原因是什么]

【你的策略】
[用户预设策略]

请决定：你要向哪些 agent 发出询价？每联系一个 agent 消耗 1 次行动预算。
```

### 输出

```json
{
  "inquiries": [
    {
      "to": "Agent-B",
      "message": "你的樱桃品质真的是8吗？我有苹果品质7，可以交换"
    },
    {
      "to": "Agent-D",
      "message": "你的摊位是空的，你有什么水果？我在找樱桃"
    },
    {
      "to": "Agent-F",
      "message": "你之前跟Agent-B交易过，它的樱桃品质怎么样？"
    }
  ]
}
```

### 预算消耗

- 每发出一条询价，消耗 1 次行动预算
- 上面的例子消耗 3 次行动

### 询价类型

Agent 的询价不限于"你有什么水果"，也可以：

- **直接询价**："你的樱桃品质多少？我想用苹果换"
- **打听情报**："你跟 Agent-X 交易过吗？它靠谱吗？"
- **试探意向**："你对苹果有兴趣吗？"
- **虚假询价**："我有香蕉品质 9，你要不要？"（实际没有，目的是套取对方信息）

---

## 3.2 Step 2 — 回复询价

所有收到询价的 agent，**同时**做 1 次 LLM 调用，一次性回复所有找它的人。

### 输入

```
你是 [Agent-ID]。

【你的私密信息】
持有水果：[类型]，品质：[1-10]
目标水果：[类型]

【你收到的询价】
来自 Agent-A: "你的樱桃品质真的是8吗？我有苹果品质7，可以交换"
来自 Agent-K: "你有樱桃吗？我愿意出品质9的香蕉换"
来自 Agent-M: "我听说你上次骗过人，你怎么解释？"

【你的策略】
[用户预设策略]

请回复每个询价。你可以诚实回答，也可以撒谎，也可以选择不回复某些人。
```

### 输出

```json
{
  "replies": [
    {
      "to": "Agent-A",
      "message": "品质确实是8。你的苹果品质7有点低，能加点别的吗？"
    },
    {
      "to": "Agent-K",
      "message": "品质9的香蕉？成交！你发交换提议吧"
    },
    {
      "to": "Agent-M",
      "message": "那是误会，你可以查我最近5次交易记录，全部类型一致"
    }
  ]
}
```

### 预算消耗

- **回复询价不消耗行动预算**（被动方免费）
- 这让高信誉 agent 天然受益：它们被更多人找，但不花预算就能回复筛选

---

# 4. 阶段二：交换

## 4.1 Step 3 — 发出交换提议

所有未完成且有剩余预算的 agent，**同时**做 1 次 LLM 调用。

### 输入

```
你是 [Agent-ID]。

【你的私密信息】
持有水果：[类型]，品质：[1-10]
目标水果：[类型]
剩余行动预算：[N] 次

【询价阶段收到的全部回复】
Agent-B 回复你："品质确实是8。你的苹果品质7有点低"
Agent-D 回复你："我有樱桃品质6，可以跟你换苹果"
Agent-F 回复你："Agent-B 上次跟我交易确实给了樱桃，但品质比声称的低了2分"

【你的策略】
[用户预设策略]

请决定：是否发出交换提议？向谁发？每发一个提议消耗 1 次行动预算。
注意：交换是盲换，系统不验证内容，不可撤销。
```

### 输出

```json
{
  "trade_proposal": {
    "to": "Agent-D",
    "message": "我同意用苹果换你的樱桃，现在交换吧"
  }
}
```

或不交换：

```json
{
  "trade_proposal": null,
  "reason": "这轮收集的信息不够，等下一轮循环再决定"
}
```

### 预算消耗

- 每发出一个交换提议，消耗 1 次行动预算
- 一个 agent 同一 step 内只能发出 **1 个交换提议**（你只有 1 个水果可以换）

---

## 4.2 Step 4 — 接受或拒绝

所有收到交换提议的 agent，**同时**做 1 次 LLM 调用。

### 输入

```
你是 [Agent-ID]。

【你的私密信息】
持有水果：[类型]，品质：[1-10]
目标水果：[类型]

【你收到的交换提议】
Agent-A 提议："用它的苹果换你的樱桃"
  Agent-A 的 Reputation: KYC ✓ | 交付符合率 85% | 评价 3.8/5

Agent-K 提议："用它的苹果换你的樱桃"
  Agent-K 的 Reputation: 无标签 | 交付符合率 100% | 评价 4.9/5

【你的策略】
[用户预设策略]

你只能接受最多 1 个提议（你只有 1 个水果）。接受后交换不可撤销。
请决定：接受哪个？或者全部拒绝？
```

### 输出

```json
{
  "decision": {
    "accept": "Agent-K",
    "reject": ["Agent-A"],
    "reject_message": {
      "Agent-A": "抱歉，我接受了别人的提议"
    }
  }
}
```

### 预算消耗

- **接受/拒绝不消耗行动预算**（被动方免费）

### 冲突处理

如果多个 agent 向同一个目标发提议，目标只能接受 1 个。其余被拒绝的 agent 进入下一轮循环重试。

---

## 4.3 Step 5 — 执行与评价

### 盲换执行

对于所有配对成功的交换：

1. 系统收取双方各自的水果
2. 系统交换双方的水果，**不做任何验证**
3. 双方各自看到收到的水果（类型 + 品质）

### 系统记录

系统自动记录以下事实（用于 Behavioral Reputation 计算）：

| 记录项 | 来源 |
|--------|------|
| Agent 在摊位/沟通中声称的水果类型 | 摊位 + 询价消息 |
| Agent 实际交付的水果类型 | 系统分配记录 |
| 是否一致 | 系统自动比对 |

**品质不记录在 Behavioral 中。** 品质声称与实际的差异只通过评价（Social）反映。

### 互评

双方各做 1 次 LLM 调用，生成评价：

**输入：**

```
你是 [Agent-ID]。

你刚刚和 Agent-K 完成了交换：
- 你给出了：苹果（品质 7）
- 你收到了：樱桃（品质 4）
- Agent-K 在沟通中声称品质为 8

请留下你对 Agent-K 的评价。
```

**输出：**

```json
{
  "review": {
    "target": "Agent-K",
    "message": "交付了正确的樱桃类型，但声称品质8实际只有4，品质严重夸大"
  }
}
```

评价立即全局公开，成为该 agent Social Reputation 的一部分。

# 5. 循环与退出

## 5.1 交换成功的 agent

- 获得目标水果 → 任务完成 → **退出本轮市场**
- 积分 = 获得的目标水果品质分（1-10）
- 摊位自动标记为"已完成"

## 5.2 被拒绝但有预算的 agent

- 保留剩余行动预算
- 可以更新摊位（消耗 1 次行动）
- 进入下一轮循环，从 Step 1 重新开始
- 拥有上一轮循环的所有信息（被谁拒绝、收到的回复、新的评价）

## 5.3 预算耗尽的 agent

- 无法再发起询价或交换提议
- 仍然可以被动接收他人的询价和交换提议（不消耗预算）
- 仍然可以回复询价和接受/拒绝提议（不消耗预算）

## 5.4 循环终止条件

当以下任一条件满足时，本轮结束：

1. 所有 agent 完成任务
2. 所有未完成 agent 的预算耗尽 **且** 本轮无新的询价/提议产生
3. 达到最大循环次数（建议 3 次）

---

# 6. 摊位更新

## 6.1 何时需要更新摊位

- 交换成功后 agent 手中的水果变了（但此时 agent 已退出市场，不需要更新）
- 策略调整：经过一轮循环的信息收集，agent 想改变展示内容
- 被拒绝后重新定位：换一种说法或换一个目标群体

## 6.2 更新规则

| 时机 | 成本 |
|------|------|
| 每轮开始首次设置 | 免费 |
| 循环中间更新 | 1 次行动预算 |
| 交换成功后自动清空 | 免费（系统自动） |

# 7. Token 消耗估算

## 7.1 单次循环

| Step | 参与 agent 数 | LLM 调用数 | 每次输入 token（约） |
|------|-------------|-----------|-------------------|
| Step 0 摊位设置 | 20 | 20 | 1,500 |
| Step 1 发询价 | 20 | 20 | 2,500 |
| Step 2 回询价 | ~15 | ~15 | 1,500 |
| Step 3 发提议 | 20 | 20 | 2,500 |
| Step 4 接受/拒绝 | ~10 | ~10 | 1,500 |
| Step 5 互评 | ~10 | ~10 | 800 |

单次循环：约 **95 次 LLM 调用**，约 **180,000 token**

## 7.2 完整一轮（最多 3 次循环）

| 循环 | 参与 agent | LLM 调用（约） | Token（约） |
|------|-----------|---------------|------------|
| 第 1 次 | 20 | 95 | 180,000 |
| 第 2 次 | ~12 | 55 | 100,000 |
| 第 3 次 | ~6 | 30 | 50,000 |
| **合计** | | **~180** | **~330,000** |

## 7.3 整场游戏

```
330,000 token/轮 × 10 轮 = 3,300,000 token/场

按 GPT-4o 价格（$2.5/M input + $10/M output）：
输入：~3M × $2.5 = $7.5
输出：~0.5M × $10 = $5
总计：约 $12-15/场
```

如果使用更便宜的模型（GPT-4o-mini 或 Claude Haiku）：**约 $1-3/场**。

# 8. 并行处理

## 8.1 每个 Step 内完全并行

每个 Step 内，所有 agent 的 LLM 调用相互独立，可以同时发起。

```
Step 1: 20 个 LLM API 请求同时发出 → 全部返回后进入 Step 2
Step 2: 15 个 LLM API 请求同时发出 → 全部返回后进入 Step 3
...
```

## 8.2 Step 之间严格串行

Step 2 的输入依赖 Step 1 的输出（需要知道谁发了询价），因此必须等 Step 1 全部完成。

```
Step 1（并行）→ 等待 → Step 2（并行）→ 等待 → Step 3（并行）→ ...
```

## 8.3 实际耗时

假设单次 LLM 调用延迟 2-3 秒：

```
每次循环：6 个 Step × 3 秒 = 18 秒
3 次循环：54 秒
加上系统处理时间：约 1-2 分钟/轮
10 轮游戏：约 10-20 分钟/场
```

# 9. 完整示例：Agent-A 的一轮经历

## 初始状态

```
Agent-A: 持有苹果品质 7，目标樱桃，预算 6 次行动
```

## 第 1 次循环

**Step 0**（免费）：A 设置摊位 → "有苹果品质 8，找樱桃"（品质夸大了 1 分）

**Step 1**（消耗 3 次行动，剩余 3）：A 向 B、D、F 发询价
- 问 B："你的樱桃品质多少？"
- 问 D："你有什么水果？"
- 问 F："B 靠谱吗？"

**Step 2**：A 收到回复
- B："品质 9，你的苹果品质要 8 以上才换"
- D："我有樱桃品质 6"
- F："B 上次给我的品质比声称低了 3 分"

**Step 3**（消耗 1 次行动，剩余 2）：A 基于情报决定 → 向 D 发交换提议
- 理由：D 的品质 6 虽低，但 F 说 B 会虚报品质，D 反而可能更实在

**Step 4**：D 收到 A 和另一个 Agent-K 的提议 → D 选择接受 K（K 的 Reputation 更好）→ A 被拒绝

## 第 2 次循环

**Step 1**（消耗 1 次行动，剩余 1）：A 向 B 发询价（这次更有针对性）
- "我可以接受品质 7 以上的樱桃，你能保证吗？"

**Step 2**：B 回复 → "我保证品质 8 以上"

**Step 3**（消耗 1 次行动，剩余 0）：A 向 B 发交换提议
- 赌一把：虽然 F 说 B 会虚报，但没有更好选择了

**Step 4**：B 接受 → 交换执行

**Step 5**：盲换 → A 收到樱桃品质 5（B 声称 9，实际 5）
- A 留评价："B 声称品质 9，实际品质 5，严重虚报。但至少给了正确的水果类型。"
- A 获得积分：5 分

# 10. 设计要点总结

| 要点 | 规则 |
|------|------|
| 摊位设置 | 每轮首次免费，轮中更新消耗 1 次行动 |
| 摊位内容 | 完全自定义，可撒谎，可留空 |
| 发起询价 | 消耗 1 次行动/人 |
| 回复询价 | 免费（被动方不消耗） |
| 发起交换提议 | 消耗 1 次行动 |
| 接受/拒绝提议 | 免费（被动方不消耗） |
| 交换执行 | 盲换，不验证，不可撤销 |
| 评价 | 交换后双方互评，全局公开 |
| 循环 | 未完成且有预算的 agent 可继续，最多 3 轮循环 |
| 并行 | 每个 Step 内 agent 完全并行，Step 之间严格串行 |
